# `tuple` 索引 —— 按位置(position)、常量下标

> ADR-0106(修复 F83)。元组的索引运算符 `t[i]` 现在能正确工作了,语义
> 与 Python 完全一致 —— **按位置寻址**,且**每个位置的类型独立正确**
> (元组是异构的,每个位置可以是不同的类型)。这是跨类型索引这条线的
> 最后一员(`str` 按码点、`bytes` 按字节、`list` 按元素、`tuple`
> 按位置)。

## 先看示例

```cobrust
fn main() -> i64:
    let t: (i64, str) = (7, "x")
    print(t[0])          # 7      (不再是那个被静默吞掉的 0)
    print(t[1])          # x
    let u: (i64, str, i64) = (1, "a", 2)
    print(u[2])          # 2
    print(u[0] + u[2])   # 3      (是真正的整数,按位置定型)
    print(t[-1])         # x      (最后一个元素,Python 的从末尾数)
    return 0
```

与 Python 一致:`(7, "x")[0] == 7`、`(1, "a", 2)[2] == 2`、
`(7, "x")[-1] == "x"`。

## 异构、按位置定型

元组各个位置可以是**不同的类型**。索引会返回**该位置的精确类型** ——
`(i64, str)[0]` 是 `i64`,`(i64, str)[1]` 是 `str`:

```cobrust
fn main() -> i64:
    let t: (i64, str, i64) = (1, "a", 2)
    print(t[0])   # 1   (i64)
    print(t[1])   # a   (str)
    print(t[2])   # 2   (i64)
    return 0
```

## 负数常量下标 `t[-1]`(Python 从末尾数)

**负数**下标从末尾往回数,与 Python 一致:`t[-1]` 是最后一个元素,
`t[-2]` 是倒数第二个。因为下标是编译期常量,这一切在**编译期**就完成
解析(并做越界检查)—— 没有任何运行时开销,也不会触发运行时陷阱。

```cobrust
fn main() -> i64:
    let t: (i64, i64, i64) = (10, 20, 30)
    print(t[-1])     # 30
    print(t[-2])     # 20
    print(t[-3])     # 10
    return 0
```

## 下标必须是**常量**

因为元组是异构的,只有当下标是**编译期常量**时才能确定元素类型。
**动态**下标没有唯一的静态类型,因此会在**编译期被拒绝**(§2.5-A
“把错误挡在编译期”原则)—— 而不是静默地读出一个类型错误的值:

```cobrust
fn main() -> i64:
    let t: (i64, i64) = (1, 2)
    let i: i64 = 1
    print(t[i])      # 编译错误:元组需要一个常量整数下标
    return 0
```

> 如果你确实需要动态索引,请改用 `list`(同构、按元素定型)——
> 诊断信息里也会这么提示。

## 常量下标越界会在编译期被拒绝

越界的常量下标 —— 无论哪个方向(`(1,2)[5]` 越过末尾,或 `(1,2)[-5]`
越过开头)—— 都会在**编译期被拒绝**(这与 `list`/`str` 的运行时索引
在运行时触发陷阱不同)。下标是常量,编译器在程序运行之前就能抓住它:

```cobrust
fn main() -> i64:
    let t: (i64, i64) = (1, 2)
    print(t[5])      # 编译错误:元组下标越界
    return 0
```

## 现在支持哪些写法

| 写法 | 结果 | 说明 |
|---|---|---|
| `t[i]`(常量 `i`) | 第 i 个位置的精确类型 | 按位置;**负**常量从末尾数(`t[-1]` 是最后一个) |
| `t[i]`(动态 `i`) | 编译错误 | 元组需要常量下标 —— 动态索引请用 `list` |
| `t[i]`(常量越界) | 编译错误 | 越界,在编译期就被抓住 |

元组**切片**(`t[lo:hi]`)暂未支持(Python 里它返回一个元组);
这是已记录的后续工作。

## 设计说明

- **F83 修复**:修复前,`(7, "x")[0]` *能正常构建*却静默打印出 `0`
  (CPython 是 `7`)—— 一个 §2.2 静默误编译。当时有两层桩代码:MIR
  下降阶段没有元组索引分支(它落到一个 no-op,读了错误的槽位),而
  LLVM 后端把元组降级成了空指针(构造**和**字段读取都未实现)。
  现在元组会降级为一个真正的结构体值,逐字段构造,`t[i]` 直接读取
  第 i 个字段。
- 这补齐了跨类型索引这条线:`str`(按码点)、`bytes`(按字节)、
  `list`(按元素)、`tuple`(按位置)现在全部索引正确。
- 这直接服务于 §2.5 的“LLM 优先”原则:LLM 会凭它的 Python 先验写出
  `t[0]` / `t[-1]`;静默的 `0` 会在最基础的元组用法上发生偏离 ——
  而动态的 `t[i]` 会在编译期被抓住,并附带一条告诉你怎么改的提示,
  而不是误编译。
