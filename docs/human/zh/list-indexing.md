# `list` 索引与切片 —— 按元素(element)寻址

> ADR-0096(修复 F81)。`list` 的索引运算符 `xs[i]` 和切片运算符
> `xs[lo:hi]` 现在能正确工作了,语义与 Python 完全一致 —— **按元素寻址**
> (list 是“按元素寻址”的成员,与“按字节寻址”的 `bytes`、“按码点寻址”的
> `str` 互为对应)。

## 先看示例

```cobrust
fn main() -> i64:
    let xs: list[i64] = [10, 20, 30]
    print(xs[0])          # 10
    print(xs[-1])         # 30     (最后一个元素,不再是 0)
    let ys: list[i64] = xs[1:3]
    print(len(ys))        # 2      (一个全新的 [20, 30])
    print(ys[0])          # 20
    print(ys[1])          # 30
    return 0
```

与 Python 一致:`[10,20,30][-1] == 30`、`[10,20,30][1:3] == [20,30]`。

## 负数标量索引 `xs[-1]`(Python 的“从末尾数”索引)

**负数标量索引**从末尾往回数,与 Python 完全一致:`xs[-1]` 是
**最后一个元素**,`xs[-2]` 是倒数第二个,依此类推。

```cobrust
fn main() -> i64:
    let xs: list[i64] = [10, 20, 30]
    print(xs[-1])     # 30
    print(xs[-2])     # 20
    print(xs[-3])     # 10
    return 0
```

> **旧行为(已被取代)。** 在 ADR-0096 之前,`xs[-1]` 会静默返回 `0`
>(F81 §2.2 缺陷)—— 任何负数索引、以及任何越界的正数索引,都会悄悄
> 给出 `0`。现在它会返回正确的元素(负数情形),或大声地 trap(越界
> 情形)。

## 越界标量索引会 trap(绝不静默返回错值)

真正越界的标量索引 —— 无论**哪个方向**(`xs[100]` 越过末尾,或
`xs[-100]` 越过开头)—— 都会在**运行时 trap**(一行清晰的
`list index out of range: i=.. len=..` 信息,退出码 3),与 Rust 自身的
切片越界 panic 一致。它**绝不会**是一个静默的 `0`。

```cobrust
fn main() -> i64:
    let xs: list[i64] = [10, 20, 30]
    print(xs[100])    # TRAP:list index out of range: i=100 len=3
    return 0
```

程序能正常**构建** —— 这个 trap 是**运行时**守卫,因为索引的具体值
只有在程序运行时才知道。

## 现在支持的形式

| 形式 | 结果 | 说明 |
|---|---|---|
| `xs[i]` | `T` | 第 `i` 个元素;**负数** `i` 从末尾数(`xs[-1]` 是最后一个) |
| `xs[lo:hi]` | `list[T]` | 切片(返回一个全新 list);元素区间 `[lo, hi)`;越界时按 Python 方式 clamp |

切片采用 Python 的 clamp 语义 —— 越界的高位边界会被收窄到长度,
反向区间得到空 list(绝不报错):

```cobrust
fn main() -> i64:
    let xs: list[i64] = [10, 20, 30, 40]
    let a: list[i64] = xs[1:99]
    print(len(a))     # 3      (高位边界被 clamp 到长度 -> [20,30,40])
    let b: list[i64] = xs[3:1]
    print(len(b))     # 0      (反向区间 -> [])
    return 0
```

每次切片都会产生一个由你的作用域**拥有**的全新 list,在作用域退出时
恰好释放一次;输入 `xs` 只是被**借用**(读取),绝不会被消耗 —— 所以
你可以对同一个 `xs` 反复切片。

## 暂不支持的切片形式(在编译期被拒绝)

只支持“两个边界都显式给出、非负、默认步长”的 `xs[lo:hi]` 形式。其它
所有形式都会在 `cobrust check` 阶段被**拒绝**(`UnsupportedSliceShape`),
而不是像以前那样静默错编译 —— 这正是 §2.5-A “在编译期捕获错误”原则,
诊断信息会打印正确写法 `xs[1:len(xs)]`:

| 形式 | 状态 |
|---|---|
| `xs[1:]` / `xs[:3]` / `xs[:]`(开放端) | 拒绝 |
| `xs[0:4:2]`(带步长) | 拒绝 |
| `xs[1:-1]`(负数边界) | 拒绝 |

支持这些形式是 ADR-0096 中列明的后续工作。在它们落地之前,请写出
显式的非负边界。

## 设计说明

- **F81 修复**:修复前,`print([10,20,30][-1])` 会静默打印 `0`(BUG 1),
  而 `let ys = xs[1:3]` 能构建、却在运行时以 `misaligned pointer
  dereference` **崩溃** —— list 切片当时是个未实现的桩,返回整数 `0`
  并把它当作 list 句柄使用(BUG 2,未定义行为)。两者现已修复,与
  CPython 3 逐字节对齐。
- **运行时**:`__cobrust_list_get`(把负数索引归一化为 `len + i`,对真正
  越界的读取 trap)与 `__cobrust_list_slice`(为 `[lo, hi)` 铸造一个全新
  list),对应 `str`/`bytes` 的索引/切片机制,但是**按元素寻址**。
- 这补齐了跨类型的索引这条线:`str`(按码点)、`bytes`(按字节)、
  `list`(按元素)现在全都索引/切片正确,具备从末尾数的负数索引、
  越界 trap、以及有界的 `lo:hi` 切片。
- 这直接服务于 §2.5 的“LLM 优先”原则:LLM 会凭它的 Python 先验写出
  `xs[i]` / `xs[-1]` / `xs[lo:hi]`;一个静默的 `0` 哨兵值、或一次 UB
  崩溃,都会在最常见的 list 习惯写法上与预期产生偏离。
