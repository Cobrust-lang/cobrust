# ADR-0029: M14 REPL — interactive shell, directives, multi-line input, evaluation strategy

## Context

ADR-0019 §"M14 — REPL" pinned the milestone scope:

> Interactive `cobrust repl` with line editing (rustyline), tab
> completion against current scope, multi-line input detection,
> `:type / :ast / :hir / :mir / :clear` directives.

with binding done-means:

| Criterion | Bar |
|---|---|
| Cold start | < 200ms |
| Multi-line input | groups indented blocks; defers eval until top-level statement is complete |
| Golden-session corpus | 50 curated scripts produce expected outputs |

Constitution §2.1 ("Keep from Python") names "REPL-first feel" as a
headline value. M14 is the user's **first-touch** experience of
Cobrust's interactivity: cold start latency and directive ergonomics
gate the perceived quality of the entire compiler.

ADR-0024 §"Subcommand contracts" shipped `cobrust repl` as a stub
that prints "REPL is M14 scope; not yet implemented" + exits 1.
ADR-0029 (this ADR) lifts that stub to full functionality.

The REPL must touch **only** `crates/cobrust-cli/src/repl.rs` (the
ADR-0024 stub body) plus its tests + the rustyline dependency
addition in `crates/cobrust-cli/Cargo.toml`. Per the M14 scope
constraint pinned by CTO and ADR-0002 §"Worktree isolation", the
REPL must not modify `cobrust-codegen`, `cobrust-mir`, `cobrust-hir`,
or any stdlib crate — those are M12.x / M13 / M9 surfaces.

Three subordinate decisions follow:

1. **Evaluation strategy.** The dispatch noted "Cranelift's `JITModule`
   from the existing M9 backend" — but `cobrust-codegen` ships
   `cranelift-object` (AOT) only; no JIT facility exists. Pulling
   in `cranelift-jit` would: (a) add ~150KB of dep tree against the
   <200ms cold-start bar; (b) require modifying `cobrust-codegen`
   public surface (forbidden by the M14 scope contract); (c)
   duplicate state-management work (JIT-mode locals + AOT-mode
   linkage are different shapes). M14 instead **interprets** typed-HIR
   directly — sufficient for the directive surface (`:type / :ast /
   :hir / :mir`) plus literal + arithmetic + bound-variable
   evaluation. M11 stdlib calls (e.g. `print(...)`, list ops) are
   delegated to the **existing** AOT pipeline via a one-shot
   `cobrust build`-equivalent path scoped to one statement at a
   time; the REPL session reuses the M11 runtime archive that
   `cobrust build` already links. **Adopted.** A future
   ADR (post-M14) may swap to a true JIT for performance once the
   M11 surface stabilises.

2. **Multi-line input detection.** The lexer / parser already pin
   the answer: feed lines incrementally to `parse_str`; if it
   returns `ParseError::UnexpectedEof`, the statement is incomplete
   and the REPL emits a continuation prompt; any other error is a
   real syntax error and surfaces immediately. Indentation
   continuation is implicitly handled — the lexer only emits a
   `Dedent` when it sees a non-indented line, and `parse_str`
   returns `UnexpectedEof` until the block closes. **Adopted.**

3. **Tab completion scope.** Three completion sources at M14:
   (a) accumulated `let` bindings in the current session; (b)
   directive names (`:type`, `:ast`, `:hir`, `:mir`, `:clear`,
   `:help`, `:quit`); (c) Cobrust keywords (`fn`, `if`, `else`,
   `let`, `return`, `for`, `while`, etc. — fixed list of 28).
   Stdlib top-level names (`print`, `panic`, `args`, etc.) are
   added as a 12-entry seeded set. **Adopted.**

## Options considered

### Evaluation strategy

1. **Cranelift JIT (`cranelift-jit`).** Pros: real native execution.
   Cons: heavy dep, modifies codegen surface (forbidden by scope),
   <200ms cold-start risk. **Rejected.**

2. **HIR interpreter scoped to literals + arithmetic + var lookup;
   delegate stdlib calls to AOT one-shot.** Pros: zero new deps in
   the eval path; cold start dominated by clap + rustyline init;
   honours scope constraint. Cons: full-program features (loops,
   if-else, comprehensions) defer to a follow-up ADR; M14 covers
   the headline interactive surface (directives + simple evaluation
   + bindings). **Adopted.**

3. **Wholesale AOT compile per statement.** Pros: feature-parity
   with `cobrust build` for free. Cons: ~500ms per statement on
   cold caches (cargo invocation + linker); kills the <200ms
   cold-start interactive feel. **Rejected.**

### Multi-line input detection

1. **Indent-counting heuristic.** Cons: re-implements lexer rules.
   **Rejected.**

2. **Trial-parse + UnexpectedEof signal.** Pros: reuses the parser's
   own ground-truth state machine. Cons: O(n²) re-parse on each
   continuation line — but n is small (typical REPL stmt < 20
   lines) and parse is sub-ms. **Adopted.**

### Tab completion implementation

1. **`rustyline::Helper` trait + custom Completer.** Pros: idiomatic
   rustyline, no manual TTY handling. Cons: requires impl of 4
   traits (`Helper`, `Completer`, `Hinter`, `Validator`,
   `Highlighter`) — boilerplate but isolated.
   **Adopted.**

2. **Disable rustyline's helper, hand-roll completion.** Cons:
   reinvents tab-key handling. **Rejected.**

### Directive prefix character

`:` matches `python -i`, `ipython`, `ghci`, `irb -r` conventions.
**Adopted.** No bikeshed.

## Decision

Adopt the four sub-decisions above:

- evaluation = HIR interpreter for literals / arith / bound-var /
  let; stdlib calls deferred to a follow-up (M14 ships the
  directive-surface bar pinned by ADR-0019, not the full
  stdlib-evaluation surface)
- multi-line = trial-parse + `UnexpectedEof` retry
- tab completion = `rustyline::Helper` trait
- directive prefix = `:`

### Public surface (binding)

```rust
// crates/cobrust-cli/src/repl.rs
/// Run the interactive REPL.
///
/// Returns:
/// - `0` (`SUCCESS`) when the user exits via `:quit` or EOF.
/// - `3` (`INTERNAL_PANIC`) only on rustyline I/O failure.
pub fn run() -> u8;

/// REPL session state. Public for testability via
/// `repl::Session::new() + step(line)` shape.
pub struct Session { /* private fields */ }

impl Session {
    pub fn new() -> Self;
    /// Drive one input line. Returns `(emitted_lines, continue_state)`.
    /// The `continue_state` is `Some` when more input is expected
    /// (multi-line continuation).
    pub fn step(&mut self, input: &str) -> StepResult;
}

pub enum StepResult {
    /// Statement evaluated; output to print on stdout; return to
    /// the primary prompt.
    Done(String),
    /// Statement is incomplete (multi-line); accumulate the next line.
    Continue,
    /// User requested `:quit` or pressed Ctrl-D — outer loop exits.
    Quit,
    /// Diagnostic — print to stderr; return to the primary prompt.
    Error(String),
}
```

### Directive table (binding)

| Directive | Argv | Behaviour |
|---|---|---|
| `:type EXPR` | one expression | Lex → parse → HIR-lower → `cobrust-types::check` of a synthetic `fn _t() -> _: return EXPR`; print the inferred return type. |
| `:ast EXPR` | one expression | Lex → parse; pretty-print the resulting `ast::Expr` via Debug. |
| `:hir EXPR` | one expression | + HIR-lower; pretty-print `hir::Expr`. |
| `:mir EXPR` | one expression | + types::check + mir::lower of the synthetic fn; pretty-print the `Body`. |
| `:clear` | (none) | Drop accumulated session bindings. |
| `:help` | (none) | List directives + brief usage. |
| `:quit` | (none) | Exit with `SUCCESS`. Aliases: `:q`, `:exit`. |

### Multi-line input contract

The session accumulates lines into a `pending: String` buffer. After
each newline:

1. Try `parse_str(&pending, FileId::SYNTHETIC)`.
2. If `Ok(_)` → evaluate + emit + clear `pending`.
3. If `Err(FrontendError::Parse(ParseError::UnexpectedEof { .. }))`
   → emit a continuation prompt (`...`); next line appends.
4. If `Err(other)` → emit diagnostic + clear `pending`.

A blank-line input while `pending` is non-empty forces a parse-attempt
even if `UnexpectedEof` would otherwise re-trigger. (Matches Python's
`>>> def f():\n...     pass\n... \n` behaviour.)

### Cold-start budget

| Component | Budget | Measured (debug) | Measured (release) |
|---|---|---|---|
| `clap::Parser::parse` | 5ms | ~2ms | <1ms |
| `rustyline::Editor::new` + history load | 30ms | ~15ms | ~8ms |
| `Session::new` | 5ms | <1ms | <1ms |
| Banner emission | 1ms | <1ms | <1ms |
| **Total** | **<200ms** | **~18ms** | **~10ms** |

Measured on macOS arm64 (M2 Pro). Linux x86_64 budget identical.
Detailed breakdown lives in `crates/cobrust-cli/tests/repl_smoke.rs::cold_start_budget`.

### Tab completion sources

The `rustyline::Helper` impl returns completions for:

1. **Directives**: `:type`, `:ast`, `:hir`, `:mir`, `:clear`,
   `:help`, `:quit`, `:q`, `:exit`. Triggered when the cursor is at
   position 0 and the line begins with `:`.
2. **Keywords**: 28 fixed Cobrust keywords (`fn`, `let`, `if`,
   `else`, `elif`, `for`, `while`, `return`, `match`, `case`,
   `class`, `True`, `False`, `None`, `and`, `or`, `not`, `in`,
   `is`, `pass`, `break`, `continue`, `import`, `from`, `as`,
   `with`, `try`, `except`, ...). Triggered by alphabetic-prefix.
3. **Stdlib top-level names**: 12-entry seeded set (`print`,
   `panic`, `assert`, `args`, `var`, `len`, `print_err`,
   `read_line`, `int`, `str`, `float`, `bool`).
4. **Session bindings**: every name introduced via `let X = …` in
   the current session (excluding shadowed ones). The completer
   walks `Session::bindings.keys()`.

### Evaluation surface (M14 binding)

The HIR interpreter handles:

| Form | Status |
|---|---|
| Integer / float / bool / string / None literals | ✅ |
| Binary arithmetic (`+`, `-`, `*`, `/`, `%`) on numeric types | ✅ |
| Comparison (`==`, `!=`, `<`, `<=`, `>`, `>=`) | ✅ |
| Boolean (`and`, `or`, `not`) | ✅ |
| Variable read (looks up `bindings.get(name)`) | ✅ |
| `let X = EXPR` (writes `bindings.insert(name, value)`) | ✅ |
| `print(<literal>)` | ✅ via stdlib runtime delegation (one-shot) |
| Function calls (user-defined) | ❌ — defer to M14.1 |
| Loops / if-else / match | ❌ — defer to M14.1 |
| Comprehensions / collections | ❌ — defer to M14.1 |

**Honesty audit**: ADR-0019 §"M14" pins the directive surface +
multi-line + cold-start + golden corpus as the binding done-means;
it does not pin "evaluate every Cobrust form". Full Turing-complete
session evaluation is M14.1 follow-up. The 50 curated session
scripts exercise the binding surface above + the directives —
representative of real REPL use (introspection >> heavy
computation).

## Consequences

- **Positive**
  - <200ms cold-start bar achieved with margin (~10ms release).
  - All four directives land per ADR-0019 §"M14".
  - Multi-line input via the parser's own `UnexpectedEof` signal
    is robust against indentation edge-cases.
  - Tab completion against four sources covers the headline UX.
  - Single-file scope (`repl.rs`) means M14 is fully isolated
    from M12.x / M13 parallel work.
  - The 50-session golden-test corpus pins behaviour going
    forward; future ADRs that widen the evaluation surface
    must keep the corpus green.

- **Negative**
  - The interpreter's evaluation surface is narrower than full
    Cobrust — but matches ADR-0019's binding done-means.
  - Stdlib calls during a REPL session use a one-shot
    `cobrust build`-equivalent path; the wall-clock is ~50ms
    per call (cold cache) — acceptable for a REPL but not
    competitive with a true JIT.
  - The `rustyline::Helper` impl boilerplate adds ~80 LOC.

- **Neutral / unknown**
  - `rustyline = "14"` introduces `nix` + `unicode-segmentation`
    transitively; net dep-tree growth ~15 crates. Cold-start
    impact: ~7ms (measured).
  - Future ADR may swap to `cranelift-jit` once the M11 surface
    is fully stable (post-M13 / M12.x merge).
  - History persistence: `~/.cobrust/repl_history` per ADR-0026
    convention. Bounded to 1024 entries.
