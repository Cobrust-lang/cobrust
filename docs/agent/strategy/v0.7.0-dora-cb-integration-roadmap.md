### 2.3 Key technical decisions (empirical from `Cargo.toml` + source)

| Concern | dora-rs choice | Evidence |
|---|---|---|
| **Transport** | Zenoh 1.8 (SHM data plane + TCP/UDP fallback) | workspace `Cargo.toml`; `zenoh = { version = "~1.8", features = ["shared-memory", ...] }` |
| **Wire format** | Apache Arrow 58 (columnar, zero serialization overhead) + bincode envelope | `dora-message/Cargo.toml`: `arrow-data`, `bincode`; `dora-node-api/Cargo.toml`: `arrow.features = ["ipc"]` |
| **Message envelope** | **bincode** (NOT Cap'n Proto, despite mandate brief assumption) | `libraries/message/Cargo.toml` line `bincode = { workspace = true }` |
| **Async runtime** | tokio 1.28 multi-thread (`rt-multi-thread`, `time`, `macros`, `sync`) | `apis/rust/node/Cargo.toml` |
| **Tracing** | opentelemetry 0.31 (optional via `tracing` feature) | `apis/rust/node/Cargo.toml`: `opentelemetry = { version = "0.31.0", optional = true }` |
| **Python binding** | PyO3 0.28 + pyarrow + maturin | `apis/python/node/Cargo.toml`; `pip install dora-rs` |
| **C binding** | Hand-written `node_api.h`; opaque void-pointer ABI | `apis/c/node/node_api.h` |
| **State store** | redb (embedded KV) for coordinator HA | README |
| **Real-time** | `--rt` CLI flag → mlockall + SCHED_FIFO; per-node `cpu_affinity` YAML | README "Production readiness" |
| **Dataflow descriptor** | YAML (`dataflow.yml`); optional type annotations via `output_types`/`input_types` | `examples/*/dataflow.yml` |

### 2.4 Node API surface (Rust, the FFI target)

From `examples/rust-dataflow/node/src/main.rs` + `apis/rust/node/Cargo.toml`:

```rust
use dora_node_api::{DoraNode, Event, EventStream, IntoArrow, dora_core::config::DataId};

fn main() -> eyre::Result<()> {
    let (node, events) = DoraNode::init_from_env()?;
    let rt = tokio::runtime::Builder::new_multi_thread().enable_all().build()?;
    let _rt_guard = rt.enter();
    run(node, events)
}

fn run(mut node: DoraNode, mut events: EventStream) -> eyre::Result<()> {
    let output = DataId::from("random".to_owned());
    for event in events.into_iter() {
        match event {
            Event::Input { id, metadata, data } => match id.as_str() {
                "tick" => {
                    let v: u64 = compute();
                    node.send_output(output.clone(), metadata.parameters, v.into_arrow())?;
                }
                _ => {}
            },
            Event::Stop(_) => break,
            _ => {}
        }
    }
    Ok(())
}
```

Surface size: ~7 core symbols (`DoraNode`, `EventStream`, `Event`, `DataId`,
`IntoArrow`, `init_from_env`, `send_output`) + a handful of metadata types.
**Small enough for full FFI coverage in a single Cobrust-side wrapper crate.**

### 2.6 Example node (Python) — minimum-viable pattern

```python
# sender.py
import pyarrow as pa
from dora import Node

def main():
    node = Node()
    for sent in range(100):
        event = node.try_recv()
        if event is not None and event["type"] == "STOP":
            break
        node.send_output("message", pa.array([sent]))

if __name__ == "__main__":
    main()
```

```yaml
# dataflow.yml
nodes:
  - id: sender
    path: sender.py
    outputs:
      - message
  - id: receiver
    path: receiver.py
    inputs:
      message: sender/message
```

`dora start dataflow.yml` boots daemon + coordinator + nodes. That's the
end-to-end UX a Cobrust node must replicate.

## 3. Cobrust capability gap audit

Each row is verified against the current workspace (HEAD `b1f1769`) by
grepping `crates/*/Cargo.toml` for the relevant dep.

| dora-rs need | Cobrust today | Gap severity | Resolution path |
|---|---|---|---|
| **tokio async runtime (multi-thread)** | YES — `cobrust-stdlib` `std.task` per ADR-0028, tokio multi-thread singleton via `OnceLock` | None | Reuse existing runtime; ensure `rt-multi-thread + time + macros + sync` features enabled |
| **PyO3 bindings** | YES, BUT pyo3 **0.22** (Cobrust) vs **0.28** (dora-rs main) | LOW — version delta only | Bump `cobrust-numpy` pyo3 to 0.28 OR pin dora-rs target version; ADR amendment |
| **Apache Arrow** | NO — zero `arrow` deps in any `crates/*/Cargo.toml` | **HIGH** — primary wire format; cannot send/recv without this | Add `arrow-data + arrow-array + arrow-schema + arrow-ipc` to a new `cobrust-arrow` crate OR vendor minimal subset |
| **bincode envelope** | NO — `serde_json` only | MEDIUM | Add bincode workspace dep; trivial integration with existing serde-derive |
| **Zenoh transport** | NO — zero `zenoh` deps | **HIGH** — required for dora-cb to participate in dataflow | Add zenoh ~1.8 to FFI wrapper crate; Zenoh handles SHM + TCP/UDP fallback transparently |
| **Cap'n Proto** | NO — but NOT NEEDED (dora-rs uses bincode, mandate-brief assumption corrected) | NONE | Strike from Y.3 scope; replace with bincode + Arrow |
| **ZMQ** | NO — but NOT NEEDED (dora-rs uses Zenoh, NOT ZMQ; mandate-brief assumption corrected) | NONE | Strike from Y.4 scope; Zenoh subsumes |
| **Shared memory** | NO direct binding — Zenoh embeds SHM transparently | None at Cobrust layer | Subsumed by Zenoh dep |
| **opentelemetry tracing** | Partial — `tracing` crate per ADR-0028 + ADR-0025; **no opentelemetry exporter** | LOW-MEDIUM | Add `opentelemetry` 0.31 exporter crate (optional feature) — fits §2.2 +§5.1 elegant doctrine as opt-in |
| **YAML descriptor parsing** | Partial — `serde_yaml` via `cobrust-tomli` adjacent; not direct dep | LOW | Add `serde_yaml` to FFI wrapper |
| **C ABI for cross-language** | YES — `cobrust-stdlib` exports staticlib + cdylib per ADR-0025 | None | Cobrust IS a C-ABI-emitting language; FFI to dora-c-node ABI is natural |
| **Real-time mlockall + SCHED_FIFO** | NO — pure tokio, no rt primitives | LOW (dora-side flag handles) | dora's `--rt` flag handles process-wide; Cobrust nodes inherit |
| **redb / coordinator-side state** | NO — but NOT NEEDED (coordinator is dora's job, not Cobrust's) | NONE | Cobrust is a node author, never the coordinator |
| **ROS2 bridge** | NO | Out of scope for v0.7.0 minimum bar | Defer |

**Gap summary**: 2 HIGH (arrow, zenoh), 2 MEDIUM (bincode envelope, opentelemetry export), 3 LOW (pyo3 bump, yaml, real-time). **Striking the mandate-brief Cap'n Proto + ZMQ assumptions reduces scope by ~30%.**

## 4. Integration architecture options

### 4.1 Option A — FFI to dora-rs Rust runtime

```
┌────────────────────────────────────────────────────────────┐
│ Cobrust source: my_node.cb                                 │
│   import dora                                              │
│   node = dora.Node()                                       │
│   for event in node: ...                                   │
└─────────────────────┬──────────────────────────────────────┘
                      │ compiles to cdylib via LLVM
                      ▼
┌────────────────────────────────────────────────────────────┐
│ cobrust-dora crate (NEW)                                   │
│   - Cobrust runtime ABI shim                               │
│   - thin Rust wrapper around `dora-node-api` 0.2.x         │
│   - Re-exports DoraNode, Event, EventStream, send_output   │
│   - Uses dora-rs's own tokio + zenoh + arrow internally    │
└─────────────────────┬──────────────────────────────────────┘
                      │ direct cargo dep
                      ▼
                 dora-node-api 0.2.x (upstream crate)
```

**Pros**:
- Reuses dora-rs's production-tested runtime, Zenoh wiring, Arrow schemas, opentelemetry exporters
- Always-current with upstream (cargo bump)
- Same FFI shape as `cobrust-numpy` PyO3 path → familiar pattern (§5.1 of CLAUDE.md, one-way-to-do-it)
- Smallest first-shipped surface: ~7 core symbols
- **End-to-end testable in days**, not weeks: write a Cobrust node, compile to cdylib, drop into `dataflow.yml`

**Cons**:
- Cobrust → dora binding is FFI, not native — debugging crosses ABI boundary
- pyo3 version delta (0.22 → 0.28) may force a Cobrust-wide upgrade or a feature-gated pyo3-0.28 path
- Arrow type system leaks into Cobrust user code (`pa.array(...)`) — feels Python-y not Cobrust-y; conflicts mildly with §2.5 LLM-first overlap principle since LLMs trained on pyarrow already produce this syntax (so actually a PRO under §2.5 maximize-overlap rule)

**§2.5 compliance**: STRONG — Python-side `dora.Node()` + `pa.array(...)` is exactly what LLMs trained on the dora-rs Python README + tutorials will produce ex ante. Maximize-overlap.

### 4.2 Option B — Full L0-L3 translation of dora-rs Python bindings

```
┌────────────────────────────────────────────────────────────┐
│ Cobrust user node: my_node.cb                              │
└─────────────────────┬──────────────────────────────────────┘
                      ▼
┌────────────────────────────────────────────────────────────┐
│ cobrust-dora crate (NEW, fully L0-L3 translated)           │
│   - Cobrust port of `dora/__init__.py` + `__init__.pyi`    │
│   - L0 spec extracted from dora pytest suite               │
│   - L1 translated function-by-function                     │
│   - L2 verified against dora Python differential harness   │
│   - L3 PyO3 wrapper validated downstream                   │
└─────────────────────┬──────────────────────────────────────┘
                      │ FFI to:
                      ▼
                 dora-node-api-c (C ABI surface)
                      │
                      ▼
                 dora-rs internals (Zenoh, Arrow, etc.)
```

**Pros**:
- Demonstrates the full CLAUDE.md §4.2 closed-loop pipeline on a 2nd
  ecosystem (tomli was the proof-of-concept; dora would be the
  industrial-grade follow-up)
- Output is "native Cobrust" — no FFI debugging at user-visible layer
- Maximizes the AI-translation investment thesis

**Cons**:
- **Much** longer wall-clock: L0 + L1 + L2 + L3 gates × ~30 Python symbols ≈ 30-80 hours of LLM Router consumption (per tomli baseline calibration)
- Translation provenance manifest required per CLAUDE.md §4.4 — full Cargo manifest header + verification artifacts
- Requires dora-rs Python testsuite running locally as oracle (added CI surface)
- Still ends with FFI to dora-c-node — the FFI boundary doesn't disappear, it just moves
- Risk: Python binding evolves faster than L0-L3 can re-verify (Python API at 0.2.x, no stability commitment yet)

**§2.5 compliance**: MEDIUM — output looks more Cobrust-native (e.g. `dora.Node()` with Cobrust type sigs), but cost-to-LLM-author ratio is high; doesn't strongly beat Option A on the maximize-overlap axis.

### 4.3 Option C — Hybrid: Cobrust runtime + bridge-only nodes

```
┌──────────────────┐     ┌──────────────────┐
│ Cobrust node A   │     │ Cobrust node B   │
└────────┬─────────┘     └─────────┬────────┘
         │   Arrow over IPC        │
         └────────────┬────────────┘
                      ▼
            ┌──────────────────┐
            │ dora-rs bridge   │  ← thin shim node, FFI-only
            └────────┬─────────┘
                     ▼  Zenoh
            ┌──────────────────┐
            │ rest of dataflow │
            └──────────────────┘
```

**Pros**:
- Maximum architectural freedom on the Cobrust side
- Cobrust-Cobrust nodes don't pay FFI cost

**Cons**:
- Build a parallel mini-dataflow runtime — far outside v0.7.0 scope
- Two coordination layers; failure modes multiply
- Conflicts with "正式参与机器人项目" mandate — the bar is Cobrust nodes inside dora dataflows, not Cobrust dataflows alongside dora dataflows
- Violates §5.1 one-way-to-do-it doctrine
- **Rejected** for v0.7.0; revisit only if Option A's FFI overhead proves unacceptable on benchmarks (it won't)

## 5. Recommendation: **Option A (FFI to dora-rs Rust runtime)**

**Verdict**: Adopt Option A for v0.7.0. Defer Option B as post-v0.7.0
research investment.

### 5.1 Rationale

| Criterion | A (FFI) | B (L0-L3) | C (hybrid) |
|---|---|---|---|
| Wall-clock to v0.7.0 ratification bar | **DAYS-WEEKS** | months | months |
| LLM Router token cost | low | high | medium |
| Maintenance burden (upstream tracks) | cargo bump | re-translate | dual runtime |
| Risk of upstream drift | low | HIGH (api at 0.2.x, no SemVer commitment) | medium |
| End-user UX | python-like Cobrust, FFI hidden | full Cobrust native | unclear |
| §2.5 LLM-first overlap | **STRONG** (pyarrow patterns) | MEDIUM | WEAK |
| §5.1 one-way-to-do-it | **PASS** | PASS | FAIL |
| Demonstrates CLAUDE.md §4.2 thesis | weakly (FFI is well-trod) | **STRONGLY** | n/a |
| Risk of cratering v0.7.0 schedule | LOW | HIGH | EXTREME |

The CLAUDE.md §4.2 thesis demonstration argument for B is real but
**dominated** by the schedule risk. tomli + dateutil already prove the
pipeline; doing it again on dora doesn't add new evidence — it just adds
risk. **Save B for a post-v0.7.0 research milestone** (call it Phase Y'
or M16) when the v0.7.0 ratification pressure is off.

### 5.2 Why Option A beats Option B on §2.5

The user's robotics-engineer LLM, when asked "write a dora node in
Cobrust", will produce:

```python
# What LLM emits (training-data overlap)
from dora import Node
import pyarrow as pa

node = Node()
for event in node:
    if event["type"] == "INPUT":
        data = event["value"].to_pylist()
        node.send_output("out", pa.array([data[0] * 2]))
```

Under **Option A**, this code compiles + runs because Cobrust `dora` is a
thin Cobrust wrapper around dora-node-api Rust crate, and `pyarrow` is a
Cobrust-pyo3-bound shim that already knows how to round-trip Arrow arrays.
First-shot success rate: high.

Under **Option B**, the LLM-trained code might not compile because the
Cobrust-native Arrow type doesn't precisely mirror `pyarrow.Array` (since
B translates Python *behavior*, not Python *surface*). First-shot success
rate: lower — defeats §2.5 entirely.

This argument is **the** dispositive one. CLAUDE.md §2.5 is constitutional
and overrides the §4.2 demonstration argument for B.

## 6. v0.7.0 minimum bar (the ratification gate)

A Cobrust author SHALL be able to:

1. Write `my_node.cb` using a `dora.Node()` API surface analogous to the
   Python binding (§2.5 — same patterns as `import dora; from dora import Node`)
2. `cobrust build --target dora-node my_node.cb` emits `my_node` (cdylib
   or executable; matching `dora-rs` Python wheel layout)
3. Reference `my_node` from a `dataflow.yml`:
   ```yaml
   nodes:
     - id: cobrust-sensor
       path: target/release/my_node
       outputs: [reading]
   ```
4. `dora start dataflow.yml` boots the dataflow with the Cobrust node
   participating
5. Cobrust node receives Arrow-IPC `Event::Input`, processes, emits
   Arrow-IPC `send_output`, exits cleanly on `Event::Stop`
6. End-to-end demo: Cobrust node consumes a tick + emits an Arrow array
   downstream Python/Rust node consumes correctly (round-trip validated)
7. CI smoke test pipes the demo through GH Actions
8. zh + en + agent docs updated per §3 doc mandate

**Stretch (NOT required for v0.7.0 ratification)**:

- ROS2 bridge from Cobrust
- Cobrust on the operator (in-process runtime) path, not just node path
- Distributed multi-machine demo
- Real-time `--rt` mode validation

## 7. Phasing (Y.1 - Y.7 sub-milestones)

Aligned with ADR-0070 §2.2 Y.1-Y.7 row labels; this section supplies the
empirical content + effort buckets.

### Y.1 — dora-rs API surface deep dive (this doc)

- **Status**: DONE-in-this-commit
- **Output**: §2 of this doc + this strategy file
- **Effort**: 1-2 days (already absorbed)

### Y.2 — Concurrency primitives audit

- **Bar**: validate Cobrust `std.task` (ADR-0028 multi-thread tokio) is
  binary-compatible with dora-node-api's tokio init pattern
  (`tokio::runtime::Builder::new_multi_thread().enable_all().build()`)
- **Concrete gate**: a Cobrust-emitted program that calls
  `DoraNode::init_from_env()` does not double-init the runtime
- **Risk**: tokio runtime conflict if both Cobrust stdlib and dora-rs
  try to install singletons; resolution likely a feature flag for
  "external-runtime-mode" on `cobrust-stdlib`
- **Effort**: 2-3 days

### Y.3 — Arrow + bincode dependency adoption

- **Bar**: new `cobrust-arrow` crate (or fold into `cobrust-dora`) with
  arrow 58 + bincode workspace deps; pyo3 conversion shims
- **Concrete gate**: round-trip `pa.array([1,2,3])` Cobrust ↔ Rust ↔ Arrow IPC bytes
- **§2.5 note**: User-facing surface should mirror `pyarrow.Array`
  literal patterns (NOT a Cobrust-native Tensor type — that's a §2.5
  violation)
- **Effort**: 3-5 days (includes new crate boilerplate)

### Y.4 — Zenoh transport binding

- **Bar**: `cobrust-dora` cargo dep `zenoh = "~1.8"` with SHM feature; no
  user-visible API (transport is dora-rs's internal concern)
- **Concrete gate**: `cargo build -p cobrust-dora --release` succeeds with
  zenoh transitively pulled
- **Risk**: zenoh build system complexity (C dependencies); test on
  macOS + Linux x86_64 + Linux aarch64 GH Actions matrix
- **Effort**: 2-4 days

### Y.5 — Real-time + opentelemetry exposure

- **Bar**: optional `--rt` and `--otel` features on `cobrust-dora`;
  pass-through to dora-node-api's own flags
- **Concrete gate**: `cobrust build --feature rt` + `--feature otel` succeed
- **Effort**: 1-2 days (mostly Cargo feature wiring)

### Y.6 — `cobrust-dora` wrapper crate authoring

- **Bar**: NEW crate `crates/cobrust-dora/` per Option A; surface API:
  - `Node::new()` → `Result<Node, DoraError>`
  - `Node::next()` → `Option<Event>`
  - `Node::send_output(id: &str, data: ArrowArray)` → `Result<(), DoraError>`
  - `Node::log_{info,warn,error}(msg: &str)`
  - `Node::dataflow_id()` → `String`
  - `Event::Input { id, data, metadata }`, `Event::Stop`, `Event::Error`
- **Source**: thin Rust over dora-node-api 0.2.x; pyo3 shim for Python
  compat; LLVM codegen surfaces it as Cobrust prelude member
- **Concrete gate**: unit tests pass; example dataflow with one Cobrust
  node runs end-to-end via `dora start`
- **§2.5 audit**: surface MUST mirror the Python `dora/__init__.pyi` §2.5
  patterns; deviations from `dora-rs` Python API are §2.5 violations and
  require explicit ADR justification

## 10. F35-sibling discipline

This strategy doc reflects empirical findings as of 2026-05-25. Specific
F35-sibling drift risks for downstream sprints:

- **API version drift**: dora-rs is at `0.2.1`; pre-v1 SemVer. Any sprint
  that adopts `dora-node-api` MUST pin the exact patch version in
  Cargo.toml + document the pinned commit SHA. F35-sibling: "as-of"
  qualifier required in commit messages.
- **pyo3 version drift**: if Y-Q1 resolves to "bump Cobrust to pyo3
  0.28", commit must NOT say "minor pyo3 update" — must say "pyo3 0.22
  → 0.28 breaking-change adoption, motivated by dora-rs main alignment".
- **Strike from scope**: Cap'n Proto + ZMQ assumptions in the dispatch
  brief are EMPIRICALLY WRONG (dora-rs uses bincode + Zenoh respectively);
  any sprint quoting those assumptions is F35-sibling adjacent.
- **F36-sibling fixture risk**: end-to-end demo tests (Y.7) must NOT be
  named after a behavior they don't actually verify; if a CI smoke test
  is named `dora-roundtrip-test` but only checks compilation, the F36
  pattern reasserts.
