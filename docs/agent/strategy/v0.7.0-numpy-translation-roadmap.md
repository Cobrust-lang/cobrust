### 1.3 Memory framing reconciliation

The `project_numpy_translation_architecture_insight` memory described
numpy translation as "30% Python wrapper translate + 50% C / 20%
Fortran NOT translated (FFI via PyO3 reverse-binding)". This was the
*alternate* path considered but NOT taken — see §4.1 below.

The **chosen** path (ADR-0012 §"Backend strategy: translate the
surface, bind the core") is:

| Layer | Strategy |
|---|---|
| numpy public Python API | Translate to Cobrust (DONE M7.0-M7.6) |
| numpy Cython glue | Translate via M6 shim (already proven on `cobrust-msgpack`) |
| numpy C core (`multiarray`, `umath`) | **Bind to Rust `ndarray = "0.16"` crate** (NOT bind to numpy's own `.so`) |
| BLAS / LAPACK | Bind to `ndarray-linalg = "0.16"` (OpenBLAS / Accelerate / netlib) |
| Random | Bind to `rand = "0.8"` + `rand_pcg` + `rand_distr` |
| FFT | Bind to `rustfft = "6"` (M7.6 per ADR-0021) |
| Complex dtype | Bind to `num_complex = "0.4"` (M7.6 per ADR-0021) |

This is materially different from the memory framing. The memory said
"keep numpy's `multiarray.so` and FFI through PyO3 reverse-binding".
The actual ADR-0012 chose a parallel Rust-native stack (`ndarray`
crate) — no dependency on numpy's compiled artifacts. Trade-off
documented inline §4.1.

## 4. PyO3 reverse-binding prerequisite status

### 4.1 The "two paths" decision (memory vs ADR-0012 reconciliation)

There were two architectural options for handling the C/Fortran 70%
of numpy:

**Path A — PyO3 reverse-binding into numpy's own .so** (memory framing):

- Keep `numpy/_core/_multiarray_umath.so` and `numpy.linalg._lapack_lite.so`
  as compiled artifacts.
- Cobrust translates only Python wrapper layer.
- Cobrust-translated functions FFI into numpy's C symbols via PyO3.
- Pros: byte-identical numerics with CPython numpy; zero LAPACK
  integration work; smaller cobrust-numpy crate.
- Cons: runtime dependency on a CPython + numpy install; "we're a
  Python successor that imports numpy" is awkward framing; perf bound
  by CPython object model at FFI boundary.

**Path B — Bind to Rust-native `ndarray` crate** (ADR-0012 chosen):

- Reimplement the `ndarray` data model + dtype dispatch on top of the
  `ndarray = "0.16"` Rust crate.
- Linalg via `ndarray-linalg` (OpenBLAS / Accelerate / Intel MKL).
- Random via `rand` family.
- FFT via `rustfft`.
- Pros: zero CPython runtime dependency; AOT compilation; no GIL;
  cleanly composes with Cobrust ownership model.
- Cons: must replicate numpy semantics in Cobrust+Rust; numerical
  agreement validated via differential testing (`@py_compat(numerical,
  rtol=1e-7)` per L2 gate per ADR-0007).

**ADR-0012 chose Path B**, and `cobrust-numpy` is built on it
(6643 LOC, all M7.x ADRs accepted). Survey conclusion: Path B is sunk
cost — flipping to Path A for v0.7.0 is not viable. v0.7.0 closes
Path B gaps; revisit Path A discussion in v0.8.0+ only if Path B fails
the differential bar on a critical surface.

### 4.2 PyO3 binding for Cobrust → Python users (still relevant)

Separate from Path A/B above: PyO3 bindings (per ADR-0011) let
**Python users** `import cobrust_numpy as np`. Status:

- `crates/cobrust-numpy/src/pyo3_bindings.rs` exists (91 LOC) — minimal
  surface gated behind `--features pyo3`.
- M6 proved the build path on `cobrust-msgpack`.
- **Gap for v0.7.0**: pyo3_bindings.rs covers ~5% of the public API.
  Each P0 item in §3.1 needs a corresponding `#[pyfunction]` wrapper.
  Mechanical; estimated ~400 LOC of glue.

`numpy_pyo3_compiles.rs` test exists in the tests/ directory — verifies
the PyO3 feature flag at least compiles. Does NOT verify behavior.

## 5. Deep-coupling design surfaces

Per memory §"Non-trivial design surfaces". Status assessment:

### 5.1 `__array_interface__` / `__array__` / `__array_function__`

- Hook protocols consumed by pandas / scipy / PyTorch / cupy /
  matplotlib for zero-copy interop.
- **Current state**: `pyo3_bindings.rs` does NOT implement these
  protocols. cobrust-numpy `Array` cannot today be silently passed to
  `torch.from_numpy()` without copy.
- **v0.7.0 gap**: implement `__array_interface__` (synchronous,
  dict-shaped) at minimum. `__array_function__` (NEP-18) is the
  expensive one — defer to v0.7.x.
- **Est LOC**: 80 (`__array_interface__` only).

### 5.2 dtype object metaclass

- Python `np.float64` is both a dtype and a scalar type. Inheritance
  hierarchy: `np.generic -> np.number -> np.floating -> np.float64`.
- **Current state**: `dtype.rs` (147 LOC) has the dtype enum but NOT
  the scalar type hierarchy. `isinstance(x, np.floating)` will not
  work on cobrust-numpy outputs from a Python consumer.
- **v0.7.0 gap**: scalar type hierarchy in pyo3_bindings.
- **Est LOC**: 200.

### 5.3 `np.frompyfunc` / `np.vectorize`

- Dynamic ufunc generation from a Python callable.
- **Current state**: NOT in cobrust-numpy.
- **v0.7.0 gap**: cobrust does NOT inherit Python's runtime-dispatch
  model; `np.frompyfunc(lambda x: x*2, 1, 1)` cannot be implemented
  cleanly without a Python interpreter at runtime.
- **Recommendation**: defer to v0.7.x; design a Cobrust-native
  `@ufunc` decorator (ADR required) that monomorphizes at AOT.

### 5.4 Buffer protocol / `memoryview`

- Python's PEP 3118 buffer protocol; numpy.ndarray exposes it.
- **Current state**: NOT exposed in pyo3_bindings.rs.
- **v0.7.0 gap**: implement `__buffer__` / `__release_buffer__` (PEP
  688 in py3.12+, PEP 3118 fallback) on pyclass.
- **Est LOC**: 120.

### 5.5 Aggregate

| Surface | v0.7.0 P0? | Est LOC | Defer to |
|---|---|---|---|
| `__array_interface__` | Yes | 80 | — |
| Scalar dtype hierarchy | Yes | 200 | — |
| `__array_function__` (NEP-18) | No | — | v0.7.x |
| `np.frompyfunc / vectorize` | No | — | v0.7.x (ADR-needed) |
| Buffer protocol | Yes | 120 | — |

**Deep-coupling P0 LOC**: ~400.

## 6. Phasing

### Option A — full M7.x in v0.7.0

Implement everything in §3.1 P0 + §3.2 P1 + §5 deep-coupling P0.

- Effort: ~1590 (P0 trans) + ~1580 (P1 trans) + ~400 (coupling) +
  ~400 (PyO3 glue) = ~3970 LOC of new translation.
- Per audit-#1 (tomli) data point: 800 LOC → 1 closed sprint cycle of
  L0-L3 strict gate. Multiply by 5 → 5 opus sprints (~2 weeks calendar).
- Risk: high — `__array_function__` and `vectorize` design surfaces
  unresolved.

### Option B — minimum viable (recommended)

Implement only §3.1 P0 + §5 deep-coupling P0 (excluding NEP-18 +
vectorize). Defer §3.2 P1 to v0.7.x.

- Effort: ~1590 (P0 trans) + ~400 (coupling P0) + ~400 (PyO3 glue) =
  ~2390 LOC.
- ~3 opus sprints (~1.5 weeks calendar).
- Plus: differential bar hardening (1000-input corpus expansion to
  match L2 gate) on existing 6643 LOC — non-trivial; ~1 sprint.
- **Recommended for v0.7.0 ratification.**

### Option C — verify-existing-only

Skip new translation. v0.7.0 numpy scope = "harden + document current
~60% surface, defer 40% gap to v0.7.1".

- Effort: ~1 sprint (hardening + docs only).
- Risk: violates user mandate "基本科学计算 (numpy) 必须 ready" if
  `np.mean(axis=...) / np.linspace / np.eye / dtype introspection` are
  considered "basic" (they are).
- **Not recommended.**

### Comparison

| Option | Calendar | LOC delta | User-mandate interpretation |
|---|---|---|---|
| A | ~3 weeks | +3970 | "fully ready" |
| **B** | ~2 weeks | +2390 | "basic scientific computing ready" ✓ |
| C | ~1 sprint | +0 | "what's there is what we ship" ✗ |

## 7. v0.7.0 scope recommendation

**Adopt Option B as Stream W in ADR-0070.**

Stream W sub-milestones:

| Sub | Description | Done means |
|---|---|---|
| W.1 | P0 translation gap closure (§3.1 items 1-9) | All 9 items translated; per-item differential corpus ≥ 200 inputs; clippy clean |
| W.2 | M7.6 FFT verification | `linalg_differential.rs` extended with `fft / ifft / rfft / irfft` corpus; ADR-0021 §"FFT" closure check |
| W.3 | Deep-coupling P0 (§5.5) | `__array_interface__` + scalar hierarchy + buffer protocol shipped; interop smoke test with CPython numpy → cobrust-numpy round-trip |
| W.4 | PyO3 wrapper expansion | All P0 surface has `#[pyfunction]` glue; `numpy_pyo3_compiles.rs` extended to verify behavior, not just compile |
| W.5 | Differential hardening | 1000-input corpus per L2 gate (per ADR-0007 §"L2 verification") on EVERY public function; numerical-tier `@py_compat(numerical, rtol=1e-7)` enforced |
| W.6 | Downstream validation | At least one consumer test: `cobrust-numpy` produces output → CPython numpy consumes via `__array_interface__` → expected output |

**Defer to v0.7.x**:
- §3.2 P1 items (`unique / histogram / pad / einsum / gradient`).
- `__array_function__` NEP-18.
- `np.frompyfunc / vectorize` (requires AOT-ufunc ADR).
- I/O (`loadtxt / savetxt / .npy`).
- Masked arrays / structured arrays / memmap.

### Acceptance gate for v0.7.0 W-stream ratification

- `cargo test -p cobrust-numpy --features pyo3` PASSES with 1000+
  tests (currently 39 test files; expand to 50+).
- Workspace `cargo check --workspace --locked` PASSES clean (already
  true).
- Differential corpus PASSES at `rtol=1e-7` (float) / bit-identical
  (int) per ADR-0007 L2 gate.
- ADRs ratified: 0013-0021 (current M7 series) + 1 new ADR (W-stream
  closure ADR; assign 0073 if Stream W proceeds).
